# 2. Use Cloudflare D1 for Edge Data

Date: 2024-05-15
Status: Accepted

## Context

Our application runs on Cloudflare Workers (Edge). Connecting to a centralized Postgres (e.g., AWS RDS) introduces significant latency (TCP handshake + roundtrip) and connection pooling issues from serverless environments.

## Decision

We will use **Cloudflare D1** (SQLite at the Edge).

## Rationale

- **Latency**: D1 is located near the worker, virtually eliminating network latency.
- **Cost**: Generous free tier and pay-per-read/write model fits our serverless architecture better than idle instances.
- **Integration**: Native binding (`env.DB`) in Workers avoids connection string/pooling management complexities.
- **Performance**: Read replication is automatic across the Cloudflare network.

## Trade-offs / Consequences

- **Constraints**: SQLite features only. No specialized Postgres extensions (PostGIS, etc.).
- **Size**: 10GB limit per database requires architectural discipline (sharding per tenant if we grow large).
- **Consistency**: Must handle eventual consistency for read replicas.

## Alternatives Considered

- **Neon / Supabase**: Good serverless Postgres options, but still incur HTTP/TCP overhead compared to native D1 bindings.
